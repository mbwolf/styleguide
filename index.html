Front-End Developer Styleguide

This document defines formatting and style rules for HTML, CSS, and JavaScript. It aims at improving code quality and consistency throughout all of our projects. All code in any code-base should look like a single person wrote it, no matter how many people contributed. In order to achieve this, the following coding standards must be followed by all developers here at meltmedia.

Global Style Rules

Spaces vs. Tabs

Always indent by two spaces, never mix tabs with spaces.

  • For easier diffs
  • Legacy code uses spaces
  • Consistency throughout all projects

HTML Style Rules

General formatting

Use a new line for every block, list, or table element, and indent every such child element. Also, all tags have to be closed.

Code readability and consistency

            
    <blockquote>
      <p><em>Space</em>, the final frontier.</p>
    </blockquote>

    <ul>
      <li>Moe</li>
      <li>Larry</li>
      <li>Curly</li>
    </ul>

    <table>
      <thead>
        <tr>
          <th scope="col">Income</th>
          <th scope="col">Taxes</th>
        </tr>
      </thead>
      <tbody>
        <tr>
          <td>$ 5.00</td>
          <td>$ 4.50</td>
        </tr>
      </tbody>
    </table>

Capitalization

All Element names, attributes, attribute values, selectors, properties, and property values (with the exception of strings) should be lowercase.

Consistency throughout all projects

            
    <!-- BAD -->
    <A HREF="/">Home</A>
            
          
            
    <!-- GOOD -->
    <a href="/">Home</a>

Trailing whitespace

Remove all trailing white spaces inside of html tags. Also, always leave a blank line at the end of your file.

Tip: configure your editor to "show invisibles". This will allow you to eliminate end of line whitespace, eliminate unintended blank line whitespace, and avoid polluting commits. Below are links to view instructions for various text editors:

Resources

  • Sublime Text 2: http://www.tbd.com
  • Textmate: http://www.tbd.com
  • Coda 2: http://www.tbd.com

Trailing white spaces can complicate diffs

            
    <!-- BAD -->
    <p>What? </p>
            
          
            
    <!-- GOOD -->
    <p>Yes please.</p>

Entity references

For all special characters use their numeric character reference.

Resources

  • html-entities.org
  • Bentity Tool
  • Consistent rendering across all browsers
  • Consistency throughout all projects

CSS Formatting Rules

Block content indentation

Indent all block content to reflect hierarchy and improve understanding.

Easier to read and helps reflect hierarchy

            
    @media (max-width: 960px) {

      .wrapper {
        color: #999;
        width: 100%;
      }

    }

Commenting

Be sure to comment your CSS. Use section separators to help organize the document. The goal is to make our code as self documented as possible.

  • Code organization
  • Maintainability
  • Consistency throughout projects
            
    /* --- [ GLOBAL HEADER ] --- */

      #header {
        margin: 0 auto;
        width: 980px;
      }

      /*
        This is a comment describing what the css
        below is doing and why it's that way.
      */
      .subnav {
        float: left;
        position: relative;
      }

Declaration order

Put declarations in alphabetical order. Ignore vendor-specific prefixes for sorting purposes.

Helps with consistency, also easy to remember and maintain

            
    #page-container {
      background: #ccc;
      -webkit-border-radius: 5px;
      -moz-border-radius: 5px;
      border-radius: 5px;
      border: 1px solid #e3e3e3;
      color: #202020;
      margin: 0 auto;
      text-align: left;
      z-index: 1;
    }

Vendor Prefix Order

Vendor prefixes should always come before the actual property. The last property should be the actual property because it will be based entirely on the spec, it will match other browsers implementations, and will be the only supported version as time goes on as vendor prefixes are removed.

Resources

  • Ordering CSS3 Properties
  • Consistency amongst projects
  • Future proof your CSS
            
    .modal {
      -webkit-border-radius: 5px;
      -moz-border-radius: 5px;
      border-radius: 5px;
      border: 1px solid #e3e3e3;
    }

Property name stops

Always add a space after the colon

Makes code easier to read

            
    <!-- BAD -->
    .title {
      font-weight:bold;
    }
            
            
    <!-- GOOD -->
    .title {
      font-weight: bold;
    }

Declaration stops

End every declaration with a semicolon

Consistency throughout all projects

            
    <!-- BAD -->
    .test {
      display: block;
      height: 100px
    }
            
            
    <!-- GOOD -->
    .test {
      display: block;
      height: 100px;
    }

Type selectors

Unless necessary (for example with helper classes), do not use element names in conjunction with IDs or classes. Avoiding unnecessary ancestor selectors is useful for performance reasons as well as makes your CSS more scalable.

Resources

  • Selector Performance: Simplifying CSS Selectors
  • Object Oriented CSS: Introduction to Object Oriented CSS

Makes styles to reliant on structure

            
    <!-- BAD -->
    ul#example { ... }
    div.error { ... }
            
            
    <!-- GOOD -->
    #example { ... }
    .error { ... }

Selector and declaration separation

Always separate selectors and declarations by new lines.

  • Code Readability
  • Can cause compiling issues for some preprocessors
            
    <!-- BAD -->
    a:hover, a:focus, a:visted {
      color: #c60;
      text-decoration: underline;
    }
            
            
    <!-- GOOD -->
    a:hover,
    a:focus,
    a:visted {
      color: #c60;
      text-decoration: underline;
    }

Rule separation

Always separate rules by new lines. Makes code easier to read.

Makes code easier to read and maintain

            
    body {
      background: #fff;
    }

    #container {
      color: #4a4a4a;
      width: 70%;
    }

Naming Conventions

Separate words in IDs and class names by a hyphen and all names should be lowercase.

For code consistency across all projects

            
    <!-- BAD -->
    .content_wrap {}
    #pageWrap {}
    #Footer
            
            
    <!-- GOOD -->
    .content-wrap {}
    #page-wrap {}
    #footer {}

Hexadecimal notation

Always use lowercase hex values, e.g., #aaa. Also, use 3 character hexadecimal notation where possible.

  • Consistency amongst projects
  • Hex colors can help optimize style sheets to their smallest size
            
    <!-- BAD -->
    #header {
      background: white;
      color: #cccccc;
    }
            
            
    <!-- GOOD -->
    #header {
      background: #fff;
      color: #ccc;
    }

JavaScript Rules

Semicolons

End all statements with a semicolon.

  • Consistency amongst projects
  • Minifying JavaScript without semicolons can be a little risky and may produce unpredictable results
            
    <!-- No semicolons -->
    var $form = $("register-form")
    $form.submit(onFormSubmit)
            
            
    <!-- Recommended -->
    var $form = $("register-form");
    $form.submit(onFormSubmit);

Always use Strict Equals

End all statements with a semicolon.

  • Avoid type coercion which can cause unpredictable results
  • This helps to maintain data type integrity throughout your code
  • In general, it's just safer
             
     <!-- Type converting equals -->
     function checkZero(value) {
       return value==0;
     }
     checkZero("0");
             
             
     <!-- Strict equals -->
     function checkZero(value) {
       return value===0;
     }
     checkZero("0");

Naming Conventions

The variable name of a jQuery object, always start with a dollar sign ("$")

Identifies the variable to be a jQuery object. It's good to make this separation as jQuery objects come with a different set of properties.

             
     <!-- Not starting with $  -->
     var slides = $(".slide");
     slides.each(function(i, slide) {
       slide.css("left", i*100);
     });
             
             
     <!-- Starting with $ -->
     var $slides = $(".slide");
     $slides.each(function(i, $slide) {
       $slide.css("left", i*100);
     });
